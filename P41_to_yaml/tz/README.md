# ТЗ: конвертер ручной DrawIO-схемы в SEAF YAML

## Контекст

В репозитории уже есть рабочий пайплайн:

- `seaf2drawio.py` — генерация DrawIO из YAML
- `drawio2seaf.py` — обратная конвертация DrawIO в YAML

Текущий `drawio2seaf.py` рассчитан на схемы, где объекты уже содержат служебные SEAF-атрибуты `schema`, `OID` и набор полей сущности. Ручная схема `result/Draft. Новый шаблон Р41 - Схема функциональной сетевой архитектуры v1.2 20231124_01 (2).drawio` этим требованиям не соответствует и поэтому не может быть корректно разобрана существующим конвертером.

При этом ручная схема по визуальной структуре и составу сущностей заметно близка к автоматически генерируемым схемам:

- присутствуют крупные контейнеры сетевых сегментов;
- присутствуют сети, сетевые устройства, сервисы и связи;
- часть названий совпадает или близка к `data/example`.

Из этого следует, что реалистична разработка отдельного конвертера, который будет не читать готовые SEAF-метаданные, а восстанавливать модель из визуальной структуры схемы, текста и правил.

## Цель работы

Разработать отдельный конвертер `manual_drawio -> SEAF YAML`, который по ручной схеме DrawIO формирует черновой набор YAML-файлов, пригодный для дальнейшей генерации схемы через `seaf2drawio.py`.

## Ожидаемый результат

На выходе должен формироваться набор YAML-файлов, близкий по структуре к `data/example`, включая как минимум:

- `dc.yaml`
- `dc_office.yaml`
- `network_segment.yaml`
- `network.yaml`
- `network_component.yaml`
- при возможности `network_links.yaml`
- при возможности `compute_service.yaml`
- при возможности `kb.yaml`, `storage.yaml`, `cluster_virtualization.yaml`, `k8s.yaml`

Дополнительно конвертер должен формировать отчёт о качестве распознавания:

- какие объекты определены уверенно;
- какие определены эвристически;
- какие не удалось классифицировать;
- какие поля были заполнены заглушками или оставлены пустыми.

## Область первой версии

Первая версия должна решать ограниченную задачу и не пытаться восстановить весь SEAF целиком.

### Входит в первую версию

- разбор XML/внутренней структуры `.drawio`;
- извлечение геометрии, текста, типов фигур, parent-child отношений и линий;
- выделение крупных областей схемы;
- распознавание сетевых сегментов;
- распознавание сетей;
- распознавание части сетевых устройств;
- восстановление связей между объектами;
- генерация черновых YAML-файлов;
- генерация отчёта по неоднозначным объектам.

### Не входит в первую версию

- универсальная поддержка произвольных DrawIO-файлов;
- полное автоматическое восстановление всех обязательных полей SEAF;
- гарантированное восстановление реальных IP, VLAN, провайдеров и других данных, если на схеме они заменены шаблонами вида `xxx.xxx.xxx.xxx/xx`, `VLAN xxx`, `название провайдера`;
- полная реконструкция `compute_service`, `k8s`, `kb`, `storage` без ручной доразметки.

## Основные ограничения

1. Ручная схема не содержит `schema` и `OID`, поэтому текущий `drawio2seaf.py` нельзя просто доработать минимально.
2. В схеме много шаблонных значений и placeholder-подписей.
3. Значительная часть семантики хранится не в атрибутах, а в:
   - тексте,
   - цвете и типе контейнера,
   - иконке,
   - относительном расположении,
   - вложенности,
   - линиях.
4. Некоторые сущности могут быть визуально похожи, но принадлежать разным SEAF-схемам.
5. Для ряда сущностей потребуется словарь правил и ручные маппинги.

## Архитектура решения

Рекомендуется делать новый конвертер отдельным модулем, а не перегружать текущий `drawio2seaf.py`.

Предлагаемая структура:

- `scripts/manual_drawio_to_seaf.py` — CLI-точка входа;
- `lib/manual_drawio_parser.py` — разбор DrawIO/XML;
- `lib/manual_drawio_classifier.py` — классификация объектов;
- `lib/manual_drawio_mapper.py` — маппинг в SEAF-сущности;
- `lib/manual_drawio_report.py` — отчёт о распознавании;
- `result/manual_reverse/` — выходные YAML и отчёты.

## Логика конвертации

### Шаг 1. Парсинг схемы

Извлечь из `.drawio`:

- все `mxCell`, `object`, `UserObject`;
- значения `id`, `parent`, `source`, `target`;
- `style`;
- `value`/`label`;
- координаты `x`, `y`, `w`, `h`.

### Шаг 2. Построение внутренней модели

Сформировать нормализованный набор узлов:

- контейнеры;
- вершины;
- линии;
- подписи;
- группы.

Для каждого узла определить:

- текст;
- геометрию;
- родителя;
- детей;
- связи;
- вероятный тип.

### Шаг 3. Выделение крупных областей схемы

По крупным контейнерам и координатам выделить логические зоны:

- офис;
- основной ЦОД;
- резервный/облачный ЦОД;
- внешние каналы;
- транспортные линии.

### Шаг 4. Классификация объектов

На основании текста, стиля, иконки и родителя классифицировать объекты в классы:

- `dc_office`
- `dc`
- `network_segment`
- `network`
- `network_component`
- `network_link`
- `compute_service`
- `kb`
- `storage`
- `cluster_virtualization`
- `k8s`
- `unknown`

### Шаг 5. Маппинг в SEAF

Для классифицированных объектов сформировать SEAF-записи:

- `title`
- `description`
- `location`
- `segment`
- `network_connection`
- `type`
- `zone`
- другие поля, которые можно восстановить уверенно

### Шаг 6. Генерация OID

Так как в ручной схеме нет готовых `OID`, конвертер должен уметь строить их по правилам:

- нормализация текста;
- транслитерация/очистка;
- добавление контекста площадки;
- добавление префикса по сущности.

Пример:

- `DMZ-PROD` -> `project.network.lan.dc01.dmz.prod`
- `Wireless Controller Guest` -> `project.network_device.wificontroller.guest`

Правила генерации должны быть вынесены в конфиг или словарь.

### Шаг 7. Отчёт

Для каждого объекта конвертер должен фиксировать:

- исходный ID DrawIO;
- распознанный тип;
- уровень уверенности;
- созданный OID;
- какие поля заполнены автоматически;
- какие поля требуют ручной проверки.

## Критерии готовности первой версии

Первая версия считается успешной, если:

1. Конвертер стабильно читает целевую ручную схему.
2. Уверенно извлекаются:
   - сетевые сегменты,
   - большая часть сетей,
   - часть сетевых устройств,
   - основные связи.
3. Формируется валидный YAML в структуре, совместимой с `seaf2drawio.py`.
4. Сгенерированный из YAML DrawIO визуально узнаваемо повторяет исходную структуру схемы.
5. Для нераспознанных объектов создаётся понятный отчёт.

## Теоретический план работ

### Этап 1. Исследовательский POC

Цель:

- доказать, что ручная схема стабильно разбирается;
- собрать JSON-представление всех узлов и связей.

Шаги:

1. Написать небольшой скрипт распаковки и чтения `.drawio`.
2. Выгрузить все элементы в промежуточный JSON.
3. Отдельно выгрузить:
   - вершины,
   - линии,
   - контейнеры,
   - подписи.
4. Проверить, как восстанавливаются parent-child отношения.
5. Проверить, какие типы `shape` и `style` повторяются.

Артефакты:

- `parsed_nodes.json`
- `parsed_edges.json`
- `style_catalog.json`

### Этап 2. Распознавание каркаса схемы

Цель:

- научиться автоматически выделять площадки и крупные сетевые сегменты.

Шаги:

1. Найти крупные контейнеры по размеру и стилю.
2. Построить правила для:
   - офиса,
   - основного ЦОД,
   - резервного/облачного ЦОД.
3. Построить правила для сегментов:
   - `DMZ`
   - `INET-EDGE`
   - `INT-WAN-EDGE`
   - `INT-NET`
   - `INT-SECURITY-NET`
   - офисных сегментов
4. Проверить, можно ли различать сегменты по цвету, положению и тексту одновременно.

Артефакты:

- `detected_locations.json`
- `detected_segments.json`

### Этап 3. Распознавание сетей

Цель:

- выделять сети внутри сегментов и маппить их в `seaf.company.ta.services.networks`.

Шаги:

1. Собрать словарь шаблонов для сетевых подписи.
2. Определить правила различения `LAN` и `WAN`.
3. Извлекать из текста:
   - название сети,
   - CIDR, если он указан явно;
   - VLAN, если он указан явно.
4. Если данные шаблонные, помечать поля как `review_needed`.
5. Привязать каждую сеть к площадке и сегменту.

Артефакты:

- `network.yaml`
- `network_recognition_report.yaml`

### Этап 4. Распознавание сетевых устройств

Цель:

- выделять роутеры, контроллеры, NGFW, WAF, точки доступа и близкие сущности.

Шаги:

1. Использовать `shape` и текст как основной сигнал.
2. Собрать таблицу:
   - shape -> вероятный тип SEAF
   - keyword -> уточнение типа
3. Восстанавливать `network_connection` из линий и близости к сетям.
4. Привязывать устройство к сегменту и площадке.

Артефакты:

- `network_component.yaml`
- `device_recognition_report.yaml`

### Этап 5. Распознавание сервисов

Цель:

- попробовать восстановить часть `compute_service`, `kb`, `storage`, `k8s`.

Шаги:

1. Отделить инфраструктурные сервисы от сетевых сущностей.
2. Выделить словари по ключевым словам:
   - `DNS`, `DHCP`, `NTP`
   - `AD`, `Mail`, `Jenkins`, `Confluence`, `BitBucket`
   - `S3`, `Storage`, `CEPH`
   - `K8s`, `Master Nodes`, `Worker Nodes`
3. Для первой версии ограничить объекты, которые распознаются уверенно.
4. Всё остальное отправлять в `unknown`/`review`.

Артефакты:

- `compute_service.yaml`
- `kb.yaml`
- `storage.yaml`
- `service_recognition_report.yaml`

### Этап 6. Генерация итогового YAML-пакета

Цель:

- собрать набор файлов, максимально похожий по структуре на `data/example`.

Шаги:

1. Сформировать `_root.yaml`.
2. Разложить сущности по файлам.
3. Удалить пустые поля.
4. Добавить поля, необходимые для `seaf2drawio.py`.
5. Сформировать итоговый отчёт по покрытию.

Артефакты:

- пакет YAML в отдельной директории;
- отчёт `summary_report.md`.

### Этап 7. Проверка через прямую генерацию

Цель:

- подтвердить, что результат пригоден для текущего генератора.

Шаги:

1. Подать сформированные YAML в `seaf2drawio.py`.
2. Получить новую `.drawio`.
3. Сравнить с ручной схемой:
   - по наличию сегментов;
   - по наличию сетей;
   - по наличию устройств;
   - по топологии связей;
   - по визуальному сходству.
4. Сформировать список расхождений.

Артефакты:

- `generated_from_manual.drawio`
- `comparison_report.md`

## Риски

- Слишком много шаблонных подписей без реальных значений.
- Несколько разных сущностей могут использовать одну и ту же фигуру.
- Часть важных объектов может быть представлена только декоративными элементами.
- В одной схеме смешаны реальные данные и шаблонные блоки.
- Для части связей нельзя однозначно восстановить SEAF-смысл только по линии.

## Практическая стратегия

Рекомендуемый подход:

1. Не пытаться сразу покрыть весь SEAF.
2. Сначала стабильно разобрать:
   - `network_segments`
   - `networks`
   - `components.networks`
3. Сразу встроить систему confidence-score и отчётов.
4. Все спорные места оставлять в состоянии “черновик + ручная проверка”.
5. Только после этого расширять покрытие на сервисы и вычислительную инфраструктуру.

## Итоговая оценка реализуемости

Разработка возможна и технически оправдана, если цель звучит так:

- “получить черновой SEAF YAML из ручной схемы и затем довести его до рабочего состояния”.

Разработка не выглядит оправданной, если цель звучит так:

- “получить полностью корректный и полный SEAF YAML без ручной проверки”.

Оптимальный результат первой итерации:

- частично автоматический конвертер;
- хороший отчёт;
- управляемый объём ручной доработки после импорта.
